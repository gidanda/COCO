# COCO 技術選定書（MVP）

## 1. 目的
本書は、COCOのMVP開発における技術スタックを定義するものである。

COCOは、グループごとに定義されたエリアをもとに、メンバー同士が現在の居場所を共有するスマホアプリである。MVPでは、位置情報、地図、招待URL、画像アップロード、認証を含む主要機能を、短期間で実装しやすい構成を重視して技術選定を行う。

---

## 2. 前提
- 本プロダクトはスマホアプリとして開発する
- MVPでは iOS / Android の両対応を目指す
- MVPでは手動更新を採用する
- MVPでは他ユーザーの詳細位置ピン表示は実装しない
- 将来的には自動更新、フレンド招待、詳細位置共有などの拡張を想定する

---

## 3. 採用技術一覧

| 項目 | 採用技術 |
|---|---|
| フロントエンド | React Native |
| アプリ開発基盤 | Expo |
| 言語 | TypeScript |
| バックエンド / DB / 認証 / ストレージ | Supabase |
| 地図表示 | React Native Maps |
| 位置情報取得 | Expo Location |
| 招待URL / ディープリンク | Expo Linking |

---

## 4. 各技術の選定理由

### 4.1 React Native
React Nativeは、iOS / Android の両方を単一コードベースで開発しやすい。COCOは位置情報、地図、日常利用を前提としたスマホアプリであり、ネイティブアプリに近い体験を比較的少ない実装コストで目指せる。

### 4.2 Expo
Expoは、React Nativeアプリ開発を効率化するフレームワークおよびツール群である。位置情報取得、ディープリンク、開発環境構築などが行いやすく、MVPの開発速度を高めやすい。

### 4.3 TypeScript
型安全性を確保しやすく、画面・API・DBの整合を保ちやすい。MVP段階でも、状態管理やAPIレスポンスの型を明確にできるため、実装ミスを減らしやすい。

### 4.4 Supabase
Supabaseを、バックエンド / DB / 認証 / ストレージ基盤として採用する。

選定理由は以下のとおり。
- PostgreSQLベースでDB設計と相性がよい
- Authによりメールアドレス認証を実装しやすい
- Storageによりユーザーアイコンやグループアイコンを保存しやすい
- MVPで必要な機能を一体的に扱えるため、自前バックエンド構築より初速が出しやすい

### 4.5 React Native Maps
地図画面はCOCOの主要画面の一つであり、エリアの確認・作成・管理を行う。React Nativeでの地図表示ライブラリとして広く使われており、MVPとの相性がよい。

### 4.6 Expo Location
ユーザーの現在地取得に用いる。MVPでは手動更新のたびに現在地を取得し、エリア判定に使う。

### 4.7 Expo Linking
招待URLによるグループ参加を実現するために用いる。未ログイン時のログイン導線や、ログイン後の参加確認画面への復帰にも利用しやすい。

## 5. 技術選定と機能要件の対応

| 機能 | 使用技術 |
|---|---|
| ユーザー登録 / ログイン | Supabase Auth |
| ユーザーアイコン / グループアイコン | Supabase Storage |
| グループ / エリア / 最新位置の保存 | Supabase (PostgreSQL) |
| 地図表示 | React Native Maps |
| エリア作成 | React Native Maps |
| 現在地取得 | Expo Location |
| 招待URL参加 | Expo Linking + Supabase |
| モバイルUI構築 | React Native + Expo + TypeScript |

---

## 6. 今回採用しない候補

### 6.1 Flutter
Flutterもスマホアプリ開発の有力候補であるが、今回はReact Native + Expoの方がMVPの立ち上がりが早く、地図・位置情報・ディープリンク周辺も進めやすいと判断したため採用しない。

### 6.2 自前バックエンド（FastAPI / NestJS など）
自前バックエンド構成は柔軟性が高いが、MVP段階では認証、DB、ストレージ、運用まで含めて実装範囲が広がりやすい。今回はSupabaseで初速を優先する。

### 6.3 Webアプリ構成
COCOは位置情報、地図、日常的な起動、将来的な自動更新を含むため、Webアプリよりスマホアプリとの相性が高い。したがって、今回はスマホアプリを前提とする。

---

## 7. 将来の拡張を見据えた補足
- MVPでは手動更新だが、将来的には自動更新を検討する
- 将来的にフレンド招待や詳細位置共有を追加する可能性がある
- 将来的に通知機能や共有粒度設定を追加する可能性がある
- 将来的に位置更新ロジックや判定ロジックの一部をより明確にサーバー側へ寄せる可能性がある

---

## 8. 結論
COCOのMVPでは、以下の技術構成を採用する。

- フロントエンド: React Native + Expo + TypeScript
- バックエンド / DB / 認証 / ストレージ: Supabase
- 地図表示: React Native Maps
- 位置情報取得: Expo Location
- 招待URL / ディープリンク: Expo Linking

この構成により、MVPに必要な主要機能を比較的少ない実装コストで構築しつつ、将来拡張にも対応しやすい土台を作る。
